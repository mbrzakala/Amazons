## 2. Web-Codegen-Scorer Baseline

**Status**: NOT APPLICABLE to this work

The `wcs-edd` tool is a full eval pipeline that generates code from prompts using an LLM (Gemini) and scores the output. It is not a static analysis tool for scoring existing code. Running `run-task.ps1 0` would:
1. Generate new code from task prompts using Gemini API calls (expensive)
2. Score that generated code, not our existing frontend
3. Require an `.env` file with API keys

This tool benchmarks prompt engineering strategies, not existing codebases. Our frontend was built by an AI agent (Cascade) following the implementation plan, not generated by the wcs-edd pipeline.

**Recommendation**: If the team wants a wcs-edd baseline, run it separately with a task prompt that describes the frontend UI requirements, then compare the generated output quality against our hand-built implementation.

## 5. Out of Scope (per §4 of the implementation plan)

The following are **explicitly out of scope** for this frontend UI implementation plan:

- **NestJS backend implementation** — no `apps/backend` project exists. The fake API layer (`FakeApiService`) serves as the demo data source. See `frontend-backend-contract.md` for the contract shapes and migration path.
- **TRIZ contradiction-matrix reasoning logic** — the actual inventive-principle lookup and contradiction resolution algorithm. The frontend displays reformulation data from the API; it does not compute TRIZ principles.
- **Second-method reasoning logic** (Six Thinking Hats or otherwise) — the frontend renders hat analysis data from the API; it does not generate hat perspectives.
- **Evaluation/scoring algorithm** — the feasibility/novelty/impact/risk scoring and recommendation selection. The frontend displays scores from the API; it does not compute them.

These need their own `agent_doc/4-backend/*`-driven plan before the frontend can point at a real API instead of the fake one.
